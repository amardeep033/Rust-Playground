# Observability — Round 1 Theory Reference

---

## 1. The Three Pillars

Observability means: can you understand what your system is doing from the outside, without changing the code?

There are three tools for this.

---

### 1.1 Logs

A log is a record that something happened at a point in time. Think of it like a diary entry your service writes to itself.

```
2024-01-15 14:32:01 INFO  user login failed  user_id=123 reason=wrong_password
2024-01-15 14:32:05 ERROR database timeout   query=select_orders duration_ms=5001
2024-01-15 14:32:09 INFO  order created      order_id=456 user_id=123 amount=299
```

Every log line has:
- a timestamp (when)
- a level (how serious)
- a message (what happened)
- fields (context — who, what, why)

**When to use logs**: when you want to understand exactly what happened during a specific request or error. You investigate logs after something goes wrong.

**The problem with logs at scale**: if your service handles 10,000 requests per second and you log everything, you have millions of lines. Searching through raw text is impossible. This is why structured logging exists — covered in section 8.

---

### 1.2 Metrics

A metric is a number measured repeatedly over time. You don't store "what happened", you store "how much / how many / how long".

Examples:
- `http_requests_total = 5241` — how many requests have happened
- `active_connections = 12` — how many connections right now
- `request_duration_seconds` — how long requests are taking

Unlike logs, metrics are cheap. You can keep them forever and plot them on a graph.

**When to use metrics**: dashboards and alerts. Metrics tell you when something is wrong, not why. You set up an alert like "if error rate goes above 1%, page the on-call engineer."

**Think of it this way**: logs answer "what happened in this one request?" — metrics answer "what is the system doing overall right now?"

---

### 1.3 Traces

A trace records the journey of a single request as it travels through multiple services.

Modern backends are not one service. One user action might touch:
```
Browser
   ↓
API Gateway
   ↓
Order Service
   ↓
Payment Service  ←→  Fraud Detection Service
   ↓
Database
```

A trace captures the full picture — every service the request touched, how long each took, and where it failed or slowed down.

Each step in that journey is called a **span**. The whole chain is the **trace**.

**When to use traces**: when latency is high or errors happen in a distributed system and you need to know exactly which service is the bottleneck.

---

### 1.4 The sentence that ties it together

> Metrics tell you **that** something is wrong.
> Traces tell you **where**.
> Logs tell you **why**.

Real workflow when an incident happens:
1. Alert fires because your error rate metric crossed a threshold
2. You open traces to see which service is failing
3. You open logs for that specific trace to read the exact error message

---

## 2. Spans and Traces — Explained Properly

This comes up in every interview. Know it cold.

### 2.1 Trace

A trace is the complete record of one request. From the moment the user's browser sends the request to the moment the response comes back — that entire journey is one trace.

Every trace has a **Trace ID** — a unique random string like `4bf92f3577b34da6a3ce929d0e0e4736`. Every service that touches this request stamps this ID on their work so you can find it all later.

### 2.2 Span

A span is one unit of work inside a trace. One service call, one database query, one function — each is a span.

Spans have a parent-child relationship. When Order Service calls Payment Service, the Payment Service span is a child of the Order Service span.

```
[Order Service span — 120ms          ]
    [validate_input — 5ms]
    [Payment Service span — 100ms    ]
        [Fraud check — 30ms]
        [Charge card — 70ms          ]  ← slow span, this is the bottleneck
    [save_to_db — 10ms]
```

By looking at this tree, you immediately know the charge card call is slow. Without traces, you'd have no idea.

A span carries:
- name (what operation)
- trace_id (which request)
- span_id (this specific operation)
- parent_span_id (who called me)
- start time, end time
- status: OK or ERROR
- attributes: key-value metadata

### 2.3 Request ID vs Trace ID vs Correlation ID

These all do similar things and people use them interchangeably. Here is the real distinction:

**Trace ID** — generated by the tracing system (like OpenTelemetry). Automatically propagated across services. Used to link spans together.

**Request ID** — generated by your application, usually at the API gateway or first service. A UUID you create yourself. You add it to every log line so you can find all logs for one request.

**Correlation ID** — a business-level identifier. For example, an `order_id` that appears in logs across multiple services and even multiple traces (create order, ship order, refund order — three separate traces but one order).

In small systems, request ID and trace ID are often the same thing. In large systems, they serve different purposes.

## 3. Metric Types

### 3.1 Counter

A counter only goes up. It counts how many times something happened. It never resets (except when the service restarts).

Example: `http_requests_total`

Every time a request comes in, you increment it by 1. After a day it might be 5,241,872. That raw number is not useful. What's useful is the rate:

> "How many requests per second over the last 5 minutes?"

Counters are perfect for: request counts, error counts, bytes sent, tasks completed.

### 3.2 Gauge

A gauge can go up or down. It measures a current state.

Example: `active_connections = 12`

One minute later: `active_connections = 8`. A connection closed.

Gauges are perfect for: active connections, memory usage, queue depth, CPU percentage.

### 3.3 Histogram

A histogram measures how values are distributed. For example, most requests finish in 50ms, a few take 200ms, and one took 2 seconds.

You can't store every individual measurement — that would be too much data. Instead, a histogram uses buckets:

```
under 10ms:   ████████████  1200 requests
under 50ms:   ██████████████████  2100 requests
under 100ms:  ████  400 requests
under 500ms:  █  80 requests
under 1000ms: ▌  12 requests
over 1000ms:  ▌  8 requests
```

From these buckets you can calculate percentiles — p50, p95, p99.

Histograms are perfect for: request latency, response size, queue wait time — anything where the distribution matters.

---

## 4. Percentiles — Why Average Latency Is Useless

Imagine your service handles 100 requests. The times are:

```
98 requests: 50ms each
2 requests:  5000ms each (5 seconds — something went wrong)
```

**Average**: `(98 × 50 + 2 × 5000) / 100 = 149ms`

Looking at the average, you'd think "fine, 149ms". But 2 users waited 5 seconds. That's a real problem hidden by the average.

Percentiles tell the real story:
- **p50** — the median. 50% of requests are faster than this. → 50ms. This is the experience of a typical user.
- **p95** — 95% of requests are faster than this. → 200ms maybe. These are the slower users.
- **p99** — 99% of requests are faster than this. → 5000ms. This is your worst users. The ones who are about to complain.

Companies set SLOs on p99, not average. Because the average hides pain.

> If your p99 is 5 seconds, one in every hundred users is having a terrible experience.

---

## 5. High Cardinality

Cardinality means: how many unique values can a label have?

Metrics use labels to add dimensions. For example, `http_requests_total` with label `status`:
- `http_requests_total{status="200"}` = 10000
- `http_requests_total{status="404"}` = 50
- `http_requests_total{status="500"}` = 12

Only 3 unique values. Low cardinality. This is fine.

Now imagine someone adds `user_id` as a label:
- `http_requests_total{user_id="1"}` = 5
- `http_requests_total{user_id="2"}` = 3
- `http_requests_total{user_id="3"}` = 8
- ... one entry per user

If you have 1 million users, Prometheus now stores 1 million separate time series for this one metric. It will run out of memory and crash.

**The rule**: labels should have a small, bounded set of values.

Good labels: `status`, `method`, `endpoint`, `region` — a handful of values.

Bad labels: `user_id`, `email`, `order_id`, `ip_address` — unbounded, millions of values.

This question comes up in almost every observability interview. Know why it's a problem, not just that it is.

## 6. SLI, SLO, SLA

These sound corporate but they're simple.

**SLI — Service Level Indicator**

The actual metric you're measuring. A number.

Example: "99.2% of requests completed in under 200ms over the last 30 days."

**SLO — Service Level Objective**

Your internal target for that metric.

Example: "We want 99.9% of requests to complete in under 200ms."

The SLO is a goal your team sets for itself. If you miss it, something needs to change.

**SLA — Service Level Agreement**

A promise you make to your customers, usually written into a contract.

Example: "We guarantee 99.5% uptime. If we breach this, you get credits."

The SLA is always more lenient than the SLO. If your SLO is 99.9%, your SLA to customers might be 99.5%. The gap is your safety buffer.

**In plain terms**:
- SLI → what you measured
- SLO → what you promised yourself
- SLA → what you promised your customer

## 7. OpenTelemetry

### 7.1 The problem it solves

Before OpenTelemetry, every monitoring vendor had their own SDK. If you used Datadog, you used their library. If you switched to Jaeger, you rewrote all your instrumentation. Switching vendors meant rewriting every piece of instrumentation in every service.

OpenTelemetry is an open standard. You instrument your code once using the OTel SDK, and you can send that data to any backend — Jaeger, Datadog, Grafana, whatever. The vendor changes. Your code does not.

### 7.2 The Three Roles in the Architecture

Every observability pipeline has three jobs. Different tools fill each role.

**Role 1 — Instrumentation (lives inside your Rust service)**

This is code inside your application that generates telemetry. It creates spans, records metrics, emits logs. It runs in the same process as your service.

Tools that do this:
- `tracing` crate — the Rust surface API for structured logs and spans
- `opentelemetry` crate — the OTel SDK that produces spans and metrics in the OTel format
- `tracing-opentelemetry` — bridges `tracing` spans into OTel spans (so you write `tracing` but get OTel output)
- `prometheus` crate — produces metrics in Prometheus format

**Role 2 — Collection / Export (the middleman)**

Your service produces telemetry. Something has to receive it, possibly transform it, and forward it to the backend. This is the collector.

Tools that do this:
- **OTel Collector** — a standalone process (not part of your service) that receives telemetry over OTLP, can filter/batch/route it, and forwards to one or more backends. Optional but common in production.
- **Prometheus** — for metrics specifically, Prometheus is both a collector (it scrapes `/metrics` from your service) and a storage backend. It pulls metrics on a schedule.

**Role 3 — Visualization / Storage (the backend)**

Where data is stored long-term and where engineers go to query, visualize, and alert.

Tools that do this:
- **Jaeger** — open source trace backend. You search traces, view span trees, find bottlenecks. Traces only.
- **Grafana** — open source visualization layer. Does not store data itself — it queries Prometheus for metrics, Loki for logs, Tempo for traces, and shows everything on one dashboard.
- **Prometheus** — stores metrics as time series. Powers alerts via Alertmanager. Queried by Grafana.
- **Loki** — stores logs (by Grafana Labs). Queried by Grafana.
- **Tempo** — stores traces (by Grafana Labs). Queried by Grafana.
- **Datadog** — a SaaS product that fills all three roles at once. You send everything to Datadog (metrics, traces, logs) and use their platform for storage, search, and dashboards. Convenient but expensive and vendor-locked.

### 7.3 How It All Fits Together

Two common setups:

**OSS Stack (self-hosted)**

```
Your Rust service
├── tracing + tracing-opentelemetry  →  spans via OTLP
└── prometheus crate                 →  /metrics endpoint (scraped by Prometheus)
         ↓
OTel Collector  (receives spans, forwards to Tempo)
         ↓
┌────────────────────────────────────────┐
│  Grafana                               │
│  ├── queries Prometheus (metrics)      │
│  ├── queries Tempo (traces)            │
│  └── queries Loki (logs)              │
└────────────────────────────────────────┘
```

**SaaS Stack (Datadog)**

```
Your Rust service
├── tracing + tracing-opentelemetry  →  spans
└── prometheus / metrics crate       →  metrics
         ↓
Datadog Agent (installed on your server — acts as collector)
         ↓
Datadog Cloud (storage + dashboards + alerts — all in one)
```

> **Mental model:** Your Rust service generates raw data. The Collector is a post office — it routes it to the right place. The backend stores it. Grafana/Datadog is the window where engineers actually look at it.

### 7.4 Where Each Tool Lives

| Tool | Lives where | Does what |
|---|---|---|
| `tracing` crate | Inside your Rust binary | Structured logs + span creation |
| `opentelemetry` crate | Inside your Rust binary | OTel-format spans and metrics |
| `prometheus` crate | Inside your Rust binary | Exposes `/metrics` endpoint |
| OTel Collector | Separate process, same server or sidecar | Receives, batches, routes telemetry |
| Prometheus server | Separate server | Scrapes metrics, stores time series |
| Jaeger | Separate server | Stores and visualizes traces |
| Grafana | Separate server | Dashboards over Prometheus + Tempo + Loki |
| Datadog | SaaS (cloud) | Everything — collector, storage, dashboards |

### 7.5 What you actually write in Rust

In practice you only touch `tracing` for your day-to-day code. You put `#[tracing::instrument]` on functions you want to trace. The bridge to OTel happens at setup time, not in every function.

```
#[tracing::instrument]   ← you write this on your functions
     ↓
tracing span created
     ↓
tracing-opentelemetry layer converts it
     ↓
OTel span sent via OTLP to Jaeger or Collector
```

You write `tracing`. You get OTel. One macro gives you structured logs and distributed traces simultaneously.

## 8. Structured Logging

### 8.1 The problem with plain text logs

This is a plain text log:
```
User 123 logged in from 10.0.0.1 at 2024-01-15 14:32
```

If you want to find all logins from IP `10.0.0.1`, you write a regex. If you want to count logins per user, you parse text. At 1 million log lines per hour, this is slow and fragile.

### 8.2 Structured logs

A structured log is a log where every piece of information is a separate key-value field:

```json
{
  "timestamp": "2024-01-15T14:32:01Z",
  "level": "INFO",
  "event": "user.login",
  "user_id": 123,
  "ip": "10.0.0.1",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736"
}
```

Now you can query: `user_id = 123`, `ip = "10.0.0.1"`, `level = ERROR`. Log aggregators like Loki, CloudWatch, and Elastic are built around this.

### 8.3 In Rust with `tracing`

Wrong (string interpolation — unstructured):
```rust
tracing::info!("processing request for user {}", user_id);
```

Right (field syntax — structured):
```rust
tracing::info!(user_id = user_id, request_id = %request_id, "processing request");
```

The second version makes `user_id` and `request_id` queryable fields, not part of the message string.

### 8.4 How logs connect to traces

Every log line should carry a `trace_id`. When a span is active, `tracing-opentelemetry` injects the trace ID into every log automatically.

This means: when an error happens, you find the trace ID in the log → paste it into Jaeger → see the full request journey. Or paste it into Loki → see every log line from that exact request.

This is the bridge between logs and traces.

## 10. Debugging a Latency Spike

> "Service latency increased by 5x. What do you do?"

This is the most common interview question for observability roles. Have a clear answer.

### 10.1 Step by step

**Step 1 — Scope it with metrics**
Open the dashboard. Which endpoint is affected? All of them or one? Did it start at a specific time? Was there a deployment? This narrows the problem before you dig.

**Step 2 — Check error rates**
Is it slow but successful, or is it slow and failing? These have different causes. Slow and failing points to a downstream dependency dying. Slow but successful points to resource exhaustion or a slow query.

**Step 3 — Find a slow trace**
Sort traces by duration. Open the slowest one. Look at the span tree. Which span is fat? That's your bottleneck. Is it your code, a downstream service, or the database?

**Step 4 — Read logs for that trace ID**
Take the trace ID from the slow trace. Search your logs for it. Read every log line from that request in order. Look for timeouts, retries, lock contention, unexpected errors.

**Step 5 — Fix and validate**
Deploy the fix. Watch the p99 histogram in real time. Confirm it drops back to baseline. Check error budget recovery.

### 10.2 Most likely interview questions

1. What is the difference between logs, metrics, and traces?
2. What is a span? What is a trace?
3. What is a trace ID and why does it matter?
4. Why use structured logging instead of plain text?
5. What is the difference between a Counter and a Histogram?
6. Why do we use p99 instead of average latency?
7. What is high cardinality and why is it a problem?
8. How do logs and traces connect to each other?
9. What is OpenTelemetry and why does it exist?
10. Walk me through debugging a latency spike.

## 11. Monitoring vs Observability

This is the most common conceptual interview question. People use the words interchangeably. They mean different things.

### 11.1 The Pain That Created the Distinction

Old-school monitoring worked like this: your team sat down, thought about what could go wrong, and set up alerts for those specific things.

- CPU above 90%? Alert.
- Disk above 80%? Alert.
- HTTP error rate above 1%? Alert.

This worked when systems were simple. One server. Known failure modes. Predictable behavior.

Then systems became distributed. 50 microservices. Unknown interactions. Failures that nobody had seen before. The monitoring dashboards went green, but users were complaining. Why? Because the failure was something nobody had thought to monitor for.

> **The core problem with monitoring: it only tells you about failure modes you anticipated in advance.**

### 11.2 Monitoring — Watching for Known Problems

Monitoring means: you define what "healthy" and "unhealthy" look like, and you watch for those states.

```
You define → "if error rate > 1%, alert"
System fires alert when → error rate hits 1.2%
You investigate → the thing you already knew to watch
```

Monitoring is reactive to known problems. It answers: **"Is this specific thing I'm watching broken right now?"**

Think of it like a car dashboard. It shows you speed, fuel, engine temperature. If the engine overheats, the light goes red. But if something else entirely fails — something the dashboard doesn't have a sensor for — the dashboard stays green while the car dies.

### 11.3 Observability — Understanding Unknown Problems

Observability means: your system produces enough data (logs, metrics, traces) that you can understand **any** state it is in, even states you never anticipated.

```
Something breaks in a way nobody expected
You ask arbitrary questions → "what was the DB query latency for user 123 at 14:32?"
You get answers from the existing data
You find the problem without knowing what to look for in advance
```

> **Key mental model:** With monitoring, you look for what you expected to break. With observability, you can investigate what actually broke, even if you never imagined it.

### 11.4 The Clear Distinction

| | Monitoring | Observability |
|---|---|---|
| What it answers | "Is X broken?" | "What is happening and why?" |
| Known vs unknown | Catches known failure modes | Can investigate unknown failures |
| How it works | Pre-defined metrics and alerts | Rich telemetry (logs + metrics + traces) |
| What you need in advance | You define what to watch | You instrument everything; questions come later |
| Example | "Alert if error rate > 1%" | "User 123 is having errors — what exactly happened to their request?" |

> **One sentence:** Monitoring watches for what you expected. Observability lets you understand what you didn't.

### 11.5 Why This Matters for the Interview

Fortanix is building an observability platform, not a monitoring platform. The distinction signals whether you understand what the product actually does.

If they ask "what is observability?" and you say "it's monitoring your system" — that's a red flag. The correct framing:

> Observability is the property of a system that lets you understand its internal state from its external outputs — logs, metrics, and traces. Monitoring is one use of those outputs. Observability is the foundation that makes monitoring (and much more) possible.

## 12. Cross-Questions

These are the comparison questions interviewers love. Know both sides, not just one.

### 12.1 Logs vs Metrics — When to Reach for Which?

Both can tell you something went wrong. But they serve different purposes.

**The confusion**: "I have logs that say errors happened. Can't I just count error logs instead of having a separate metric?"

Technically yes. But:
- Logs are high volume. Counting them in real time is expensive.
- Log storage is not optimized for aggregation — it's optimized for search.
- Metrics are designed for aggregation. Prometheus can compute a rate over millions of data points in milliseconds. A log query over millions of lines takes seconds.
- You can set an alert on a metric in one line. Setting an alert on a log query requires a complex pipeline.

| | Logs | Metrics |
|---|---|---|
| Good for | "What happened in this specific request?" | "How is the system behaving overall?" |
| Volume | High — one entry per event | Low — aggregated numbers |
| Query speed | Slow at scale | Fast |
| Alerting | Possible but complex | Native, simple |
| Storage cost | High | Low |
| Retention | Days to weeks | Months to years |

> **Rule:** Metrics tell you something is wrong. Logs tell you why. Don't use one to replace the other.

### 12.2 Logs vs Traces — When to Reach for Which?

Both are per-request. What's the difference?

**Logs** are flat. They tell you what happened inside one service, at one point in time.

```
14:32:01 INFO  order-service  payment request sent   order_id=456
14:32:03 ERROR order-service  payment timed out      order_id=456 duration_ms=2001
```

You can read the story of what happened inside `order-service`. But you cannot see what happened inside `payment-service` during those 2 seconds.

**Traces** are the complete picture across all services.

```
[order-service: 2050ms                              ]
    [validate_order: 10ms]
    [payment-service: 2000ms                        ]   ← the 2 seconds are here
        [fraud-check: 20ms]
        [charge-card: 1980ms  ← TIMEOUT             ]
    [notify-user: 5ms]
```

From the trace you immediately see: the bottleneck is `charge-card` inside `payment-service`. The logs from `order-service` alone would never show you this.

| | Logs | Traces |
|---|---|---|
| Scope | One service, one event | Full request across all services |
| Good for | Error details, what exactly happened | Finding bottlenecks, understanding request flow |
| Cross-service | No | Yes — that's the whole point |
| When to use | After you found the slow span in a trace | To find which service/span is the problem |

> **The workflow:** Metrics alert → Traces find the slow span → Logs explain why that span was slow.

### 12.3 Counter vs Gauge vs Histogram — The Real Difference

People memorize the names but mix up when to use them.

**Counter: things that accumulate**

Use a counter when the number only goes up and you care about the total or rate.

- Total HTTP requests: `http_requests_total`
- Total errors: `errors_total`
- Total bytes sent: `bytes_sent_total`

The raw number is rarely interesting. What matters is the rate: "how many per second over the last 5 minutes?" Prometheus computes this with `rate()`.

> If you restart your service, the counter resets to 0. Prometheus handles this automatically.

**Gauge: a current state that goes up and down**

Use a gauge when you care about the current value right now.

- Active connections: `active_connections = 12`
- Memory in use: `memory_bytes = 450000000`
- Items in a queue: `queue_depth = 47`

One minute later it might be 8, 430000000, or 52. The value changes in both directions.

**Histogram: the distribution of a value across many events**

This is the most commonly misunderstood one.

Imagine you want to track request latency. The naive approach: use a gauge and set it to the latest request's duration. But with 1000 requests per second, this gauge updates 1000 times per second and you only ever see the most recent one. Useless.

A histogram captures every request. Internally it counts how many requests fell into each bucket:

```
< 10ms:  1200 requests
< 50ms:  2100 requests
< 100ms: 400 requests
< 500ms: 80 requests
< ∞:     8 requests
```

From this you can compute p50, p95, p99. This is the right tool for latency.

> **Rule:** If you're measuring "how long did something take" or "how big was something" — use a Histogram, not a Gauge.

| | Counter | Gauge | Histogram |
|---|---|---|---|
| Direction | Only up | Up and down | Buckets accumulate |
| What it answers | How many total? What's the rate? | What is the current value? | What's the distribution? p99? |
| Example | requests, errors, bytes | connections, memory, queue depth | latency, response size |
| Common mistake | Using for current state | Using for latency | Picking wrong bucket sizes |

### 12.4 Push vs Pull for Metrics

There are two ways for a metrics backend like Prometheus to collect data.

**Pull (Prometheus default)**

Prometheus comes to your service every 15 seconds and scrapes a `/metrics` endpoint. Your service just exposes the data. Prometheus decides when to collect it.

```
Prometheus ──scrapes every 15s──→ your service /metrics
```

Good: simple, stateless on the service side, easy to debug (just curl /metrics yourself).

Bad: your service must be alive when Prometheus scrapes. Short-lived jobs (cron jobs, batch processors) finish in 5 seconds — Prometheus never catches them.

**Push**

Your service pushes metrics to a receiver (Pushgateway, OTel Collector) whenever it has data to report.

```
your service ──pushes on completion──→ Pushgateway ──→ Prometheus
```

Good: works for short-lived jobs, works in environments where Prometheus can't reach your service (private VPCs, air-gapped).

Bad: more moving parts, you have to manage the Pushgateway as a separate service.

> **Rule:** Long-running services → Pull. Short-lived jobs → Push.

### 12.5 ERROR vs WARN — The Log Level You'll Get Wrong in Production

This sounds trivial but it is a real operational problem.

**ERROR** means: something failed that requires human attention. Someone should investigate this, possibly right now.

**WARN** means: something unexpected happened, but the system recovered on its own. Worth knowing about. Investigate when you have time.

**The mistake teams make:**

Logging too many things at ERROR because "it sounds serious". Six months later, ERROR alerts fire 200 times a day. Engineers start ignoring them. When a real critical ERROR fires, it gets missed. A customer reports an outage before the team notices.

This is called **alert fatigue** — and it's caused by misusing log levels.

```
Wrong:
ERROR "User not found"          ← This is 404. Expected. Not an error.
ERROR "Invalid input"           ← This is 400. The user made a mistake, not us.
ERROR "Retry attempt 1 of 3"   ← A retry is normal behavior.

Right:
WARN  "User not found"
WARN  "Invalid input received"
INFO  "Retry attempt 1 of 3"
ERROR "All retries exhausted, payment failed"  ← This actually needs attention.
```

| Level | When to use |
|---|---|
| ERROR | Requires human action. Something failed that the system could not recover from. |
| WARN | Unexpected but the system handled it. Investigate when you have time. |
| INFO | Normal business events. Request handled. Job started. Order created. |
| DEBUG | Internal state for development. Not in production by default. |
| TRACE | Very fine-grained. Almost never in production. |

> **Rule:** If seeing 100 ERROR logs per hour feels normal, your ERROR level is broken. ERRORs should be rare and actionable.

### 12.6 Request ID vs Trace ID — The Question That Trips People Up

Both are identifiers for a request. Here is the actual difference.

**Request ID**

Generated by your application — usually at the API gateway or the first service that receives the request. It's a UUID you create with `uuid::Uuid::new_v4()`.

You put it in every log line. You return it in the response header as `X-Request-Id` so the client can include it in a bug report.

It's a simple string. Your application controls it.

**Trace ID**

Generated by the tracing system (OpenTelemetry, Jaeger, etc.). It follows a specific format (32 hex characters). It is automatically propagated to every downstream service via HTTP headers.

You don't create it manually — the OTel SDK generates it when a new trace starts.

**Why you need both**

In a small system, they're the same thing — you pick one and use it everywhere. In a larger system:

- The trace ID lives in the tracing system. It links spans across services. Jaeger knows about it.
- The request ID lives in your application logs and is returned to the client. Your support team gives it to the customer: "include this ID in your ticket."

Sometimes they're the same value. Sometimes they're different. Either way, both should appear in every log line.

| | Request ID | Trace ID |
|---|---|---|
| Who creates it | Your application code | The tracing SDK |
| Format | Any UUID | 32 hex chars (OTel standard) |
| Purpose | Correlate logs, give to client | Link spans across services in tracing backend |
| Where it lives | Your logs and response headers | Tracing system (Jaeger, Tempo) |
| Automatically propagated | No — you pass it manually | Yes — OTel SDK handles this |
